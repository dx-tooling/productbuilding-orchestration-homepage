Highlights, nach Säulen sortiert

Jede Fähigkeit gehört zu einer der drei Säulen, die zusammen ausgeliefert werden: dem Slack-Projektmanager-Agenten, dem Coding-Agenten und den automatischen PR-Previews.

Der Slack-Projektmanager

Intent-Routing

Ein LLM-gesteuerter Router klassifiziert jede Anfrage und reicht sie an den richtigen Spezialisten weiter: Issue-Ersteller, Delegator, Researcher, Kommentator, Closer oder Event-Erzähler. Jeder Spezialist hat einen knappen Prompt und ein eng zugeschnittenes Werkzeug-Set, was das Verhalten vorhersagbar hält.

Rückfragen vor Aktion

Wenn eine Anfrage vage ist, hakt der PM mit konkreten Fragen nach, statt zu raten. Erst wenn der Umfang klar ist, legt er ein Issue an oder delegiert die Arbeit — das heißt, Entwickler bekommen am Ende gut formulierte Aufgaben statt Einzeiler-Tickets.

Phasen-Tracking pro Workstream

Jede Konversation trägt eine Lebenszyklus-Phase — Intake, offen, in Bearbeitung, Review, Überarbeitung, fertig, abgebrochen. Derselbe Satz („sieht gut aus“) heißt während des Intakes etwas anderes als nach einer fertigen Preview; die Phase erlaubt dem Router, das eindeutig zu deuten.

Zwei-Spuren-Benachrichtigungspuffer

Statusmeldungen (CI grün, Preview bereit, PR geschlossen) laufen über eine „latest-wins“-Spur; menschliche Kommentare über eine Spur, die alles anhängt. Das Resultat: Der Thread bleibt auch unter einer Welle von Ereignissen lesbar, und nichts, was jemand geschrieben hat, geht still verloren.

Der Coding-Agent

Delegation an OpenCode

Sobald der PM genug Scope hat, übergibt er die Arbeit an den Coding-Agenten, indem er eine /opencode-Anweisung postet. Der Agent liest das Issue, arbeitet in einem Branch und öffnet den PR — ohne dass ein Entwickler das Issue zuerst aufgreifen muss.

Issue-Erstellung mit Duplikat-Erkennung

Bevor ein neues Issue angelegt wird, durchsucht der PM die bestehenden offenen Issues, um kein Duplikat zu erzeugen. Findet er eine nahe Übereinstimmung, verknüpft er die neue Konversation mit der bestehenden Arbeit, statt eine parallele Spur aufzumachen.

PR-Erstellung über das CI-Framework

Ein kleiner GitHub-Actions-Workflow im Ziel-Repo lauscht auf den /opencode-Kommentar des PMs, legt einen Branch an, lässt den Agenten laufen und öffnet den PR, sobald er fertig ist. Aus Slack-Sicht sieht das einfach so aus, als wäre der PR aufgetaucht.

Feedback-Relay

Wenn der Reviewer in Slack Änderungen erbittet, übersetzt der PM das Feedback in eine konkrete Anweisung an den Coding-Agenten und postet sie zurück am PR. Der Agent iteriert; die Preview aktualisiert sich; der Zyklus läuft, bis der Reviewer zufrieden ist.

PR-Previews

Stabile Preview-URLs

Jeder PR wird unter pr-{number}-preview.{domain} deployed. Die URL bleibt für die gesamte Lebensdauer des PRs gleich, sodass dieselbe Slack-Nachricht Reviewer auch nach fünf Änderungsrunden auf die neueste Version zeigt.

Voller Lebenszyklus

Previews durchlaufen pending → building → deploying → ready bei neuen Commits und werden gelöscht, sobald der PR geschlossen wird. Container, Volumes und Routing-Labels werden automatisch aufgeräumt — keine langlebigen Preview-Friedhöfe.

Preview-Vertrag pro Repo

Jedes Ziel-Repo deklariert über .productbuilding/preview/config.yml, wie seine Previews laufen sollen: Compose-Services, Health Checks, Datenbank-Migrationen, Post-Deploy-Befehle. Der Orchestrator liest den Vertrag; das Repo bleibt für seine eigene Deploy-Form verantwortlich.

Selbstheilende Reconciliation

Beim Start gleicht der Orchestrator seinen SQLite-Zustand mit Docker und GitHub ab. Wurde eine Preview nur halb abgeräumt oder ein PR geschlossen, während das System offline war, holt er das ohne manuelles Eingreifen nach.

Unter allen drei

Multi-Provider-LLM-Unterstützung

Der PM läuft standardmäßig mit Claude von Anthropic und nutzt optional einen OpenAI-kompatiblen Fallback-Provider (Fireworks, Together usw.) für mehr Robustheit. Request-Timeouts, Retry-Anzahl und Token-Budgets sind alle per Env-Variable konfigurierbar.

Eine-Box-Infrastruktur

Eine einzige EC2-Instanz betreibt den Orchestrator und Traefik — mit automatischem Wildcard-TLS. Das OpenTofu/Terraform-Modul im Repo macht daraus ein wiederverwendbares Deploy für euren eigenen AWS-Account.
