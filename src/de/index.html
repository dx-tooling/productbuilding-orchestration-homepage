ProductBuilder

Ein agentisches Setup, das
nicht-technische Teammitglieder
von der Idee zum guten PR führt

Holt euch einen KI-Projektmanager in euren Slack — wie der eine Ansprechpartner, den ihr von einer externen Entwicklungsagentur bekommen würdet. Er nimmt Anfragen entgegen, klärt den Umfang, briefed den Coding-Agenten, lässt eine Live-Vorschau des Ergebnisses entstehen und holt Feedback ein. All das, ohne dass eure nicht-technischen Teammitglieder je GitHub öffnen müssen — und ohne dass für jede kleine Änderung Entwickler einspringen müssen.

Wie eine Agentur engagieren. Ohne die Agentur.

Stellt euch vor, ihr engagiert eine externe Entwicklungsagentur als verlängerte Werkbank für euer internes Software-Team. Diese Agentur würde einen Projektmanager auf euer Konto setzen — den einen Ansprechpartner, mit dem eure Produktleute reden. Ihr würdet diesen PM in euer eigenes Slack einladen, und er kümmert sich von dort aus um alles Weitere für euch.

ProductBuilder ist dieser PM — plus die ganze Maschinerie dahinter. Nicht-technische Teammitglieder beschreiben in Slack einfach, was sie wollen. Der PM fasst die Aufgabe zusammen, briefed einen Coding-Agenten, der den Code tatsächlich schreibt, deployt eine Live-Preview des Ergebnisses, leitet Feedback hin und her, bis alle zufrieden sind, und übergibt den PR fertig zur Entwickler-Review.

Die Low-Level-Details — GitHub-Issues, Pull Requests, OpenCode-Läufe, Preview-Systeme — bleiben vollständig gekapselt. Euer Produktteam verlässt Slack nie. Eure Entwickler bekommen gut umrissene PRs statt endlosem Hin und Her zu Spezifikationen. Die Aufgabe des Systems endet bei einem guten PR; das finale Review und der Merge bleiben bei einem menschlichen Entwickler — da, wo sie hingehören.

Drei Säulen, ein Setup

Der gesamte Ablauf — Anfrage, Code, Review — wird gemeinsam ausgeliefert. Jede Säule trägt einen bestimmten Teil der Schleife.

Der Slack-Projektmanager

Eure nicht-technischen Teammitglieder erwähnen @ProductBuilder in Slack. Der PM hört zu, stellt Rückfragen, wenn die Anfrage vage ist, und entscheidet, was zu tun ist — ein Issue eröffnen, Arbeit delegieren, eine Frage beantworten oder etwas Erledigtes schließen.

Er verfolgt jede Konversation über ihren ganzen Lebenszyklus, sodass Folgefragen am richtigen Workstream landen, statt von vorn zu beginnen.

Der Coding-Agent

Sobald der PM die Anfrage umrissen hat, übergibt er die Arbeit an den mitgelieferten Coding-Agenten. Der Agent liest das Issue, schreibt den Code und öffnet einen Pull Request gegen euer Ziel-Repository.

Einfache Produkt-Änderungen müssen nie auf dem Schreibtisch eines Entwicklers landen — die Schleife von Anfrage zu PR läuft von selbst, während der PM den Fortschritt in Slack mit erzählt.

Powered by OpenCode, aufgerufen über einen CI-Workflow, den das Setup neben dem Orchestrator installiert.

PR-Previews

Jeder Pull Request, den der Agent (oder ein Entwickler) öffnet, bekommt eine Live-Preview unter einer stabilen URL — automatisch hinter Traefik deployt und bei jedem Commit aktualisiert.

Nicht-technische Reviewer klicken einen Link, probieren die Änderung wirklich aus und geben Feedback im selben Slack-Thread. Die Preview baut sich selbst ab, sobald der PR geschlossen wird.

Eine Anfrage, lückenlos verfolgt.

1

In Slack fragen

Slack-PM

Ein Produkt-Teammitglied schreibt @ProductBuilder bitte Dark-Mode-Unterstützung hinzufügen. Der PM stellt fehlende Rückfragen — ganze App oder nur ein Bereich? OS-Präferenz oder manueller Toggle? — und legt ein gut umrissenes GitHub-Issue an, in dem der geklärte Kontext schon eingearbeitet ist.

2

Der Coding-Agent übernimmt

Coding-Agent

Der PM delegiert die Arbeit. Der Coding-Agent liest das Issue, schreibt den Code und öffnet einen Pull Request. Das Teammitglied verlässt Slack nicht — der PM erzählt den Fortschritt mit, während er passiert.

3

Eine Live-Preview erscheint

PR-Preview

Der PR triggert ein automatisches Deploy hinter Traefik. Innerhalb weniger Minuten wirft der PM die Preview-URL in den Slack-Thread — „Hier ausprobieren: pr-52-preview.example.com.“ Jeder neue Commit aktualisiert dieselbe URL.

4

Reviewen, iterieren, übergeben

Zurück zum PM

Das Teammitglied gibt in Slack Feedback. Der PM gibt die Änderungen an den Coding-Agenten weiter, der pusht Updates; die Preview aktualisiert sich jedes Mal. Sobald das Teammitglied zufrieden ist, wird der PR als bereit markiert — das finale Review und den Merge übernimmt weiterhin ein Entwickler.
